Agentic QA Boilerplate

AI-driven quality engineering on Playwright + KATA + TypeScript.

A skill-based practice that pairs your codebase with an AI agent. The agent reads a structured knowledge layer, plans tests, executes them, files bugs, automates regressions, and emits release verdicts — with human checkpoints between every stage.

1.The fundamental idea

This repository is an AI-driven QA machine. You bring a project under test (the target: a frontend, a backend, or both) and the boilerplate brings the instructions, the knowledge layer and the tooling that an AI agent uses to do quality engineering against it. The repo never runs alone — it is always paired with one specific target codebase.

For the machine to work, three families of moving parts cooperate: Skills orchestrate long, multi-phase AI work; Commands perform one-shot utilities; and Scripts are what you run yourself in the terminal to keep the configuration substrate healthy. All three feed off the same fuel — a system of variables defined in .agents/project.yaml that resolves to your target's specifics at AI session bootstrap.

The whole machine in one sentence

Three kinds of moving parts (Skills · Commands · Scripts) plus one variable substrate (.agents/project.yaml) — that is the entire mental model. The rest is detail.

Read agentic-quality-engineering.md for the strategy behind the practice and context-engineering.md for why the knowledge layer is split the way it is. This page focuses on how the pieces connect — what to invoke, in what order, and what each piece actually does.

2.Three categories of moving parts

Everything you can invoke in this repo belongs to one of these three categories. The mental model holds end-to-end:

Type What it is Where it lives Who invokes it
Skill
Multi-file markdown that orchestrates the AI through long, multi-phase tasks (subagents, decisions, references). .claude/skills/<name>/SKILL.md + references/ The AI auto-activates when your prompt matches the skill's triggers. Force-load via /skill-name.
Command
Single-file markdown for one-shot, focused utilities. .claude/commands/<name>.md You explicitly type /<name>. No auto-trigger.
Script
TypeScript CLI that runs in the terminal. scripts/*.ts and cli/*.ts You run bun run <script> directly.

The three categories share the same fuel: the variable substrate in .agents/project.yaml (and Jira metadata in .agents/jira-fields.json + .agents/jira-workflows.json + .agents/jira-required.yaml). Skills resolve {{VAR}} and {{jira.*}} placeholders against it on every AI session; commands use the same resolver; scripts read the YAML and write back to it. See §5 Circuit 1 for the full resolution flow.

3.The lifecycle: five phases

Each phase has a clear trigger, a deterministic output and a frequency. You move forward when the previous phase's output exists; you loop back to Maintenance whenever documentation drifts from reality.

One-time setup on the left, recurring work on the right. Maintenance feeds back into Daily QA.

Click each phase to expand:

0 Bootstrap — install the foundation
Trigger
bun install → bun run pw:install → bun run agents:setup → bun run jira:sync-fields → bun run jira:sync-workflows → bun run jira:check. Or, if you cloned skills à la carte, run /framework-core init first.
Produces
A populated .agents/project.yaml, the CLAUDE.md → AGENTS.md symlink, the foundation scripts (agents-setup, agents-lint, sync-jira-fields, sync-jira-workflows, check-jira-setup) wired into package.json, the workspace catalog at .agents/jira-fields.json (custom fields), and the workflow catalog at .agents/jira-workflows.json (statuses + transitions per work_type).
When to repeat
One time per repo clone. If you cloned the full repo, the foundation is already in place — only bun install, bun run agents:setup, and the two jira:sync-* commands are needed. 
bun install
bun run pw:install
bun run agents:setup          # interactive walkthrough
bun run jira:sync-fields      # catalog custom fields
bun run jira:sync-workflows   # catalog statuses + transitions per work_type
bun run jira:check            # verify both catalogs against the manifest
1 Discovery — reverse-engineer the target
Trigger
/project-discovery — give it the target repo path or describe the stack.
Produces
.context/PRD/, .context/SRS/, .context/mapping/business-data-map.md, .context/mapping/business-feature-map.md, .context/mapping/business-api-map.md, .context/master-test-plan.md. The skill orchestrates the four /business-*-map commands and /master-test-plan as part of its 4-phase flow (Constitution → Architecture → Infrastructure → Specification).
When to repeat
Once per target. Re-run when the target undergoes a major architecture change. 
/project-discovery
2 Adapt — wire KATA to the target stack
Trigger
/adapt-framework. Runs strictly AFTER Phase 1 — it consumes .context/.
Produces
tests/components/api/*.ts, tests/components/ui/*.ts, tests/components/TestFixture.ts, config/variables.ts, and api/schemas/ wired to the target's auth model and stack. Two strict sub-phases: Plan (no writes, presents the diff) → user approval → Implement (writes).
When to repeat
Once per target. Re-run only when the target's auth or stack changes meaningfully. 
/adapt-framework
3 Daily QA — the four workflow skills
Trigger
Pick the skill that owns the current stage:
  • /sprint-testing — Stages 1-3 per ticket: planning, execution, reporting.
  • /test-documentation — Stage 4: ROI-driven test case documentation in the TMS.
  • /test-automation — Stage 5: Plan → Code → Review pipeline for KATA tests.
  • /regression-testing — Stage 6: regression / smoke / sanity, classification, GO/NO-GO.
Produces
ATPs, ATRs, QA comments, bug reports, TMS Test/TestPlan/TestExecution artefacts, automated tests, regression dashboards.
When to repeat
Every sprint. Every ticket. Designed to be invoked many times per week.
4 Maintenance — fix drift as it appears
Trigger
When .context/, AGENTS.md, or .agents/ drifts from reality (the target shipped new modules, new Jira fields, new endpoints).
Produces
Refreshed memory files and re-cataloged Jira fields:
  • /refresh-ai-memory — re-syncs AGENTS.md + README.md.
  • /business-data-map, /business-feature-map, /business-api-map — regenerate the maps.
  • /master-test-plan — regenerate the master test plan.
  • bun run jira:sync-fields — re-catalog Jira custom fields.
  • bun run jira:sync-workflows — re-catalog Jira workflows (statuses + transitions) when a status is renamed, a transition is added, or a workflow is swapped on an issue type. Use --force to re-prompt for already-mapped slugs.
  • bun run jira:check — verify the Jira workspace still satisfies jira-required.yaml (covers BOTH custom-field validation against jira-fields.json and work_types validation against jira-workflows.json).
  • bun run lint:agents — verify every {{VAR}}, {{jira.<slug>}}, {{jira.work_type.*}}, {{jira.status.*}} and {{jira.transition.*}} placeholder still resolves.
When to repeat
As needed. Treat drift like compiler warnings — fix them when they appear, not in batches.

4.The three confusing pieces

Three pieces have similar-sounding names and overlapping verbs ("init", "adapt", "refresh"). They are not interchangeable. Use this table to avoid running the wrong one.

Piece When you invoke it What it does Frequency
/framework-core init
(skill, active mode) 		Once, ONLY if the foundation files are missing (à la carte install). If you cloned the full repo, you NEVER invoke it directly. Generates AGENTS.md, .agents/project.yaml, .agents/jira-required.yaml, .agents/jira-fields.json, .agents/jira-workflows.json, the scripts/agents-*.ts + sync-jira-*.ts CLIs, merges entries into package.json, creates the CLAUDE.md symlink. Idempotent — existing files are preserved. One-time
/adapt-framework
(command) 		Once per target, AFTER /project-discovery finishes. Adapts tests/components/, config/, api/schemas/ to the target's stack: fixtures, page objects, API clients. Modifies THIS repo only — never the target. Plan → user approval → Implement. Once per target
/refresh-ai-memory
(command) 		Periodically, when .context/ drifts and AGENTS.md no longer reflects the project state. Re-syncs AGENTS.md and README.md against current .context/ and package.json. Touches project-specific FACTS only — never structural sections. Periodic
Mnemonic: framework-core installs the BASE, adapt-framework wires the TESTS, refresh-ai-memory updates the DOCS.

A passive role you never invoke

framework-core has a second life beyond init: it hosts shared references that workflow skills cite on demand. When /sprint-testing, /test-documentation, /test-automation, or /regression-testing delegates to a subagent, the AI loads framework-core/references/briefing-template.md, dispatch-patterns.md, and orchestration-doctrine.md automatically. You never type /framework-core to read those files — they are pulled in as part of orchestration. This passive role is invisible from the user's seat, but it is why framework-core exists even after Phase 0 is complete.

5.How everything connects internally

Two circuits explain the wiring. Circuit 1 is about variable resolution (the static substrate). Circuit 2 is about subagent orchestration (the dynamic execution model).

Circuit 1 — Variables

Variables flow: you fill the YAML once → AI reads it on every session → placeholders resolve to real values before any tool call.

.env vs .agents/project.yaml — two separate systems

Do not conflate them. They have different consumers, different lifecycles, and only one of them is committed.

.env .agents/project.yaml
Purpose Playwright + KATA runtime secrets and config. AI context-engineering variables for {{VAR}} resolution.
Consumers Test runner (bun run test, config/variables.ts, fixtures, login helpers, tests/utils/jiraSync.ts). AI agents (Claude Code, Cursor, Codex, Copilot, OpenCode) when resolving skill / template / doc references.
Examples LOCAL_USER_EMAIL, STAGING_USER_PASSWORD, XRAY_CLIENT_SECRET, HEADLESS, DEFAULT_TIMEOUT. PROJECT_KEY, WEB_URL, API_URL, JIRA_URL, DB_MCP, default_env.
Secrets? Yes (passwords, tokens, API keys). No — must remain commit-safe.
Committed? Gitignored. .env.example is the committed template. Committed.
Lifecycle Edited per developer / per CI runner. Edited once when adopting the boilerplate; rarely changes after.

The variable substrate has three syntaxes that look similar but resolve from different files. The full contract lives in .agents/README.md; the short version:

  • {{VAR_NAME}} → resolves from .agents/project.yaml (project-level static config).
  • <<VAR_NAME>> → session variable, computed at runtime by the calling skill (e.g. <<ISSUE_KEY>> extracted from the git branch). Never persisted.
  • {{jira.<slug>}} → portable Jira custom-field reference, resolves through .agents/jira-required.yaml (manifest) + .agents/jira-fields.json (workspace catalog). Sub-forms: {{jira.<slug>.<option>}} (option value for plain option / array-of-option fields), {{jira.<slug>.<parent>.<child>}} (cascading option-with-child).
  • {{jira.work_type.<slug>}} / {{jira.status.<work_type>.<slug>}} / {{jira.transition.<work_type>.<slug>}} → portable Jira workflow references. work_type resolves to the literal issue-type name (e.g. "Story"); status resolves to the literal status name (sub-keys: .id, .category); transition resolves to the transition id (sub-key: .name) — use the id form to invoke transitions unambiguously via REST. All resolve through .agents/jira-required.yaml work_types: (manifest) + .agents/jira-workflows.json (workspace catalog).

Circuit 2 — Orchestration

Each workflow skill declares its own dispatch points; the canonical briefing template + dispatch patterns live in framework-core/references/ so subagents load them once, fresh.

This is the canonical pattern the four workflow skills follow. The orchestrator reads its own dispatch strategy, writes a 6-component briefing for each subagent (Goal · Context docs · Skills to load · Exact instructions · Report format · Rules), and the subagent loads tool skills (/acli, /xray-cli, /playwright-cli) to actually run shell commands. This is what keeps the main conversation lean — the heavy reading happens inside subagents.

7.Skills catalog

Every skill that ships in .claude/skills/. Some you invoke directly; others (the tool skills and the foundation) load automatically when a workflow skill needs them.

Categories — Foundation hosts shared references and bootstraps the repo · Workflow drives a stage of the pipeline · Generator reverse-engineers context from the target · Tool wraps a CLI/MCP for I/O · Reference is loaded passively by other skills.

framework-core Foundation

/framework-core init

Bootstraps the foundation files (AGENTS.md, .agents/, scripts, package.json entries) AND hosts shared references (briefing template, dispatch patterns, orchestration doctrine) that other skills cite on demand. Two roles, one skill.

project-discovery Generator

/project-discovery

Onboards a target repo via a 4-phase flow (Constitution → Architecture → Infrastructure → Specification). Produces PRD, SRS, business-data-map, business-feature-map, business-api-map, master-test-plan.

sprint-testing Workflow

/sprint-testing

Stages 1-3 per ticket: planning, execution, reporting. Produces ATP, ATR, QA comment, bug reports. Single-ticket or batch-sprint mode.

test-documentation Workflow

/test-documentation

Stage 4. ROI-driven test case documentation in the TMS (Jira/Xray). Four scopes — module, ticket, bug, ad-hoc. Emits Candidate / Manual / Deferred verdicts.

test-automation Workflow

/test-automation

Stage 5. Plan → Code → Review pipeline for KATA + Playwright + TypeScript automated tests. Three planning scopes (module, ticket, ATC). Enforces KATA compliance.

regression-testing Workflow

/regression-testing

Stage 6. Runs regression / smoke / sanity through CI/CD. Classifies failures (REGRESSION / FLAKY / KNOWN / ENVIRONMENT / NEW TEST), computes pass-rate trends, emits GO / CAUTION / NO-GO verdict.

acli Tool

/acli

Atlassian CLI wrapper for Jira Cloud — work items, projects, boards, sprints, fields. Resolves [ISSUE_TRACKER_TOOL] pseudocode and the Jira-native [TMS_TOOL] branch (Modality B).

xray-cli Tool

/xray-cli

Xray Cloud test management via bun xray: tests, executions, plans, JUnit/Cucumber result imports, project backup/restore. Resolves [TMS_TOOL] pseudocode in Modality A.

playwright-cli Tool

/playwright-cli

Browser automation CLI: screenshots, tracing, video recording, session management, request mocking, test generation. Resolves [AUTOMATION_TOOL] pseudocode.

Note on adapt-framework

It looks like a skill (it has a slash trigger) but it ships as a single-file command at .claude/commands/adapt-framework.md. See §8 Commands.

8.Commands & Scripts

One-shot utilities — neither orchestrate long flows like skills nor maintain config like setup scripts. You type them by name.

Commands

Single-file markdown at .claude/commands/<name>.md. Invoke with /<name>.

  • /adapt-framework Adapt KATA architecture (tests/, api/schemas/, config/) to a discovered target. Plan → user approval → Implement.
  • /refresh-ai-memory Re-sync AGENTS.md, README.md (and per-tool memory files like CLAUDE.md, GEMINI.md) against current .context/ + package.json.
  • /business-data-map Generate or refresh .context/mapping/business-data-map.md — entities, flows, state machines.
  • /business-feature-map Generate or refresh .context/mapping/business-feature-map.md — feature catalog, CRUD matrix, integrations.
  • /business-api-map Generate or refresh .context/mapping/business-api-map.md — auth model, critical endpoints, architecture.
  • /master-test-plan Generate or refresh .context/master-test-plan.md — what to test and why.
  • /break-down-tests Plain-English breakdown of automated tests for a given module / spec.
  • /fix-traceability Repair broken US ↔ ATP ↔ ATR ↔ TC traceability links in the TMS.
  • /commit-push-pr End-to-end git flow: commit → push → open a PR using the test-automation template.
  • /fix-git-conflict Walk through the resolution of a merge or rebase conflict.

Scripts

TypeScript CLIs declared in package.json. Run with bun run <name>.

  • bun run agents:setup Interactive walkthrough that fills .agents/project.yaml.
  • bun run lint:agents Validate every {{VAR}}, {{jira.<slug>}}, {{jira.work_type.*}}, {{jira.status.*}} and {{jira.transition.*}} reference resolves against the manifest + catalogs.
  • bun run jira:sync-fields Catalog all custom fields from the live Jira workspace into .agents/jira-fields.json (incl. option values for option and cascading option-with-child fields).
  • bun run jira:sync-workflows Catalog Jira workflow statuses + transitions per work_type (Story, Bug, Test) into .agents/jira-workflows.json. Interactive on first run; idempotent thereafter.
  • bun run jira:check Verify the Jira workspace satisfies .agents/jira-required.yaml — covers both the custom-field catalog and the work-type / workflow catalog.
  • bun run env:validate Validate .env for the active TEST_ENV.
  • bun run api:sync Sync OpenAPI spec from the target API and regenerate api/schemas/ types.
  • bun run api:types Regenerate TS types from api/openapi.json via openapi-typescript.
  • bun run kata:manifest Extract ATCs from the codebase into a manifest. kata:manifest:watch for live watch mode.
  • bun run pw:install Install Chromium with system deps via Playwright.
  • bun xray Xray TMS sync: import / export / sync test results, executions, plans.
  • bun run test / test:e2e / test:integration / test:smoke Run the Playwright suite — full, E2E project, integration project, or @critical-tagged smoke.
  • bun run test:allure Generate and open the Allure report from ./allure-results.

Run bun <script> --help on any script for usage. The Allure binary is installed locally — no global install needed.

9.TL;DR mental model

  • Skills = play scripts for the AI (multi-act, with characters that delegate).
  • Commands = single-act utilities you type by name.
  • Scripts (bun run …) = what YOU run in the terminal to keep the fuel (.agents/project.yaml + Jira fields) healthy.
  • .context/ = what the AI KNOWS about the target (generated by /project-discovery, kept fresh by /refresh-ai-memory and the /business-*-map commands).
  • tests/ = the real code, adapted by /adapt-framework and authored via /test-automation.

Fill project.yaml first (bun run agents:setup), then let the AI discover the target with /project-discovery, then wire the tests with /adapt-framework. From there, invoke the four workflow skills depending on the QA stage you are in.
